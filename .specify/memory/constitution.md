# Overmind Constitution

## Core Principles

### I. Markdown as the Substrate

Durable artifacts — memory, references, contracts, playbooks, and active
project notes — MUST be plain Markdown on disk. Agent and human readers
share the same surface; there is no separate database or proprietary store.

**Rationale**: Markdown is greppable, diff-friendly, agent-ingestible, and
survives tool churn. The "code" of this workspace is its directory structure
and the Markdown it holds; treating prose as the durable substrate keeps
context portable across agents, models, and editors.

### II. Vendor-Neutral Contracts

`AGENTS.md` is the single canonical entry contract for any coding agent.
Vendor-specific entry files (`CLAUDE.md`, `.cursorrules`, `GEMINI.md`,
others) MUST be symlinks to `AGENTS.md` or build artifacts regenerated by
the launcher — never hand-edited duplicates. New tools MUST NOT introduce
vendor-specific configuration at the repo root when a symlink to an
existing file would suffice.

**Rationale**: Forking the contract per vendor produces drift; one source
of truth keeps every backend (Claude Code, Gemini CLI, Codex, Cursor,
Aider, future entrants) reading the same instructions.

### III. Public / Private Separation

Secrets, identity, customer/org names, and in-flight private work MUST
NOT enter this repository's git history. The enforced split:

- Workspace-scoped secrets live in `.git-ignored/secrets.env` (mode 0600).
- Private evolutionary memory lives in `memory/private/`, which is a clone
  of a separate private git repository — never committed to overmind.
- Local-only scratch (transient artifacts, experiments) lives under
  `.git-ignored/` and is never `git add -f`'d into the index.

Public memory under `memory/*.md` and `memory/<topic>/` MUST be safe to
publish. When in doubt about whether content is public-safe, default to
private placement and route through the `memory-reflect` skill's
sensitivity gate.

**Rationale**: This repo is published. Mixing public conventions with
private context — or with credentials — leaks blast radius from one
mistake (a force-add, a paste into the wrong file) into permanent history.
The separation is structural, not advisory.

### IV. Progressive Disclosure

Files MUST live where their lifetime and disclosure scope match:

- **Always-loaded** → top-level `memory/*.md` (and top-level
  `memory/private/*.md` when mounted). These are concatenated into the
  system prompt on every session and MUST stay focused; the `memory-defrag`
  skill targets 15–25 files per tier.
- **On-demand reference** → nested `memory/<topic>/...` (e.g.
  `playbooks/`). Agents navigate to these when relevant; they are NOT
  auto-loaded.
- **Project-scoped** → `projects/<name>/`. Each project may carry its own
  `AGENTS.md`; overmind's conventions do NOT transitively apply.
- **Transient** → `.git-ignored/`.

New files MUST be placed in the directory whose lifetime matches their
purpose; promoting a file from on-demand to always-loaded MUST be a
deliberate decision, not a default.

**Rationale**: An ever-growing system prompt is the same anti-pattern as
an ever-growing class. Auto-loaded context has a budget; nested directories
are how the workspace scales without paying for every byte on every turn.

### V. Validated Executable Tools

Any executable script that an agent may invoke lives under `tools/` and
MUST pass the language-appropriate CI gates before merge:

- TypeScript (Bun): lint, type-check, test.
- Python (uv): lint, pyright strict, test.
- Bash: shellcheck.

Tools MUST NOT be checked in without these gates wired up; the README and
`tools/README.md` are authoritative for the current matrix. Ad-hoc scripts
that bypass `tools/` belong in `.git-ignored/` until they are ready to be
validated and committed.

**Rationale**: Markdown can drift cheaply; executable code that an agent
runs cannot. Anything an agent invokes is a supply-chain surface for the
user's machine, so the bar for committing it is enforced by CI, not
discipline.

## Workspace Constraints

- **Project boundaries**: Work inside `projects/<name>/` defers to that
  project's own `AGENTS.md` (or equivalent). Overmind's principles apply
  to the overmind workspace itself, not transitively to embedded clones,
  submodules, or worktrees.
- **Git worktrees**: Work on any `projects/<name>/` repository SHOULD be
  done in a worktree under `$OVERMIND_ROOT/.git-worktrees/<slug>` on a
  feature branch — never directly on the project's default branch.
  Worktrees MUST be cleaned up (`worktree remove` + `worktree prune`)
  when the work merges or is abandoned.
- **Public-repo discipline**: Overmind itself is public. Changes MUST go
  through a feature branch and pull request; direct pushes to `main` are
  prohibited.
- **Generated artifacts**: Files regenerated by the launcher (e.g.
  `GEMINI.md`) are gitignored build artifacts and MUST NOT be hand-edited
  or committed.

## Development Workflow

- **Edit before create**: Prefer editing existing Markdown over adding new
  files. New top-level memory files raise the auto-loaded prompt cost for
  every future session and require the same scrutiny as a new dependency.
- **Skill design**: Skills ship under `.agents/skills/<name>/SKILL.md` in
  the open Agent Skills format. Per-skill mutable state lives in
  `.git-ignored/` or in private memory — never in the skill directory
  itself. Prefer ship-defaults-plus-markdown-overrides over imperative
  state-mutation subcommands.
- **Spec Kit lifecycle**: Feature work in this repo (when scoped through
  Spec Kit) flows constitution → spec → clarify → plan → tasks → implement,
  with the `before_*` / `after_*` hooks declared in `.specify/extensions.yml`
  driving auto-commits and feature-branch creation. The Constitution Check
  in `plan-template.md` MUST consult Principles I–V before Phase 0.
- **Commit hygiene**: Keep commits small and topical. Prefix with the
  affected directory when useful (`memory: …`, `tools: …`, `projects: …`).
  Never skip hooks (`--no-verify`) or bypass signing without explicit
  user authorization.

## Governance

- This Constitution supersedes any conflicting convention in README,
  `AGENTS.md`, memory files, or skill prompts. Conflicts MUST be resolved
  by amending one or the other so the workspace remains internally
  consistent.
- **Amendment procedure**: Amendments are proposed by editing this file
  through `/speckit-constitution`, opening a pull request, and merging via
  the same review process used for any other change. The Sync Impact
  Report at the top of this file MUST be updated in the same commit.
- **Versioning policy** (semver for governance documents):
  - **MAJOR**: Backward-incompatible removal or redefinition of a
    principle, or removal of a section.
  - **MINOR**: New principle or section added, or material expansion of
    an existing principle.
  - **PATCH**: Clarifications, wording, typo fixes, non-semantic
    refinements that do not change governance behavior.
- **Compliance review**: Pull requests touching `memory/`, `tools/`,
  `.agents/skills/`, `scripts/`, or repo-root configuration MUST verify
  alignment with the relevant principles in their description or review
  comments. Reviewers MAY block on unjustified violations; justified
  violations MUST be recorded in the affected feature plan's Complexity
  Tracking section.
- **Runtime guidance**: For day-to-day operating conventions (navigation,
  skills inventory, launcher behavior), defer to `AGENTS.md`. This
  Constitution governs *what must remain true*; `AGENTS.md` describes
  *how the workspace currently operates*.
